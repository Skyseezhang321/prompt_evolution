# CLAUDE.md

本文件供 Claude Code 在 `prompt_evolution` 仓库中工作时读取。项目目标是研究 prompt 优化与 prompt 自进化，因此所有改动都应服务于可度量、可复现、可回滚的研究闭环。

## 核心原则

1. 先定义问题，不先写 prompt
   - 在新增 prompt、实验、脚本或结论前，明确目标、假设、输入样本、模型、评估方式和成功标准。
   - 如果任务存在多种解释，先说明歧义和取舍；会影响研究结论时，先向用户确认。

2. 最小实验优先
   - 优先构建能验证一个假设的最小 prompt、eval、数据集或脚本组合。
   - 不提前引入复杂框架、通用抽象或未要求的可配置能力。

3. 一次只改一个变量
   - 尽量分开修改 prompt、数据、模型参数、评估指标、上下文组织和分析口径。
   - 如果必须同时修改多个变量，要在文档或运行记录中明确标注，避免伪因果结论。

4. 每个结论必须有证据
   - 结论要能追溯到实验记录、样例输出、指标表、失败案例或论文来源。
   - 明确区分观察、推测和结论；不要把单次样例表现写成稳定规律。

5. 目标驱动执行
   - 将“优化一下 prompt”改写成可验证目标，例如“在 baseline X 上，将指标 Y 从 A 提升到 B，同时失败类型 Z 不恶化”。
   - 多步骤任务要给出简短计划，并为每一步配套验证方式。

6. 变更可追踪
   - 重要变更记录 prompt 版本、模型、参数、数据集、评分器、成本、失败样例和回滚点。
   - 对研究追踪有意义的新增、调整和修复要写入 `CHANGELOG.md` 的 `Unreleased`。

7. 精准修改
   - 只修改当前任务需要的文件和内容，匹配现有文档与代码风格。
   - 发现无关问题可以记录或提醒，不要混入本次改动；只清理自己改动造成的无用内容。

## 工作流约束

- 新论文笔记放在 `docs/paper_notes/`，使用 `docs/paper_notes/template.md`。
- 新实验先更新 `docs/experiment_plan.md`，再实现脚本或运行实验。
- 每个 prompt 变体都要记录模型、参数、数据集、评分器、成本和失败案例。
- 自动改 prompt 的实验必须保留原 prompt、候选 prompt、优化原因、评测结果和回滚点。
- 所有消息通知统一调用 `scripts/wecom_notify.py`；真实 webhook 只能放在本地 `.env`，不得提交。
- 修改 Python 代码时运行相关测试；仅修改文档时无需强行运行测试，但要检查链接和术语一致性。
