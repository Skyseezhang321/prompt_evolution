# 别让 AI 自己改作业

> 一个创业团队学会「驯服提示词」的故事 —— 小说体科普，改写自 [跨渠道综合报告 v3](./analysis_report_v3_20260610.html)。
>
> **读前说明**：故事里的人、公司、工单、对话全是**虚构**，用来把道理讲得好玩；故事里出现的**论文数字是真的**（来自报告引用的真实研究），文末附「故事 ↔ 真实证据」对照表，较真的读者可以顺着它回到原报告和论文笔记。

## 楔子：一句让老郭皱眉的话

云梯科技只有十一个人，却接了一个看起来很体面的活：给一家连锁家电品牌做客服 AI。

产品叫「小蜜」。用户在公众号里吐一句槽，小蜜要在半秒内把这条工单读懂，吐出三样东西：

- **intent**（用户想干嘛：投诉 / 咨询 / 退款……）
- **entities**（涉及哪些订单、哪些产品）
- **urgency**（急不急）

抽对了，工单自动派给对的人；抽错了，一个想退货的客户被丢进「咨询」队列，三天没人理，然后上微博。

这天早会，老板 Diana 把一张图甩在大屏上：小蜜上线两周，准确率卡在 82%，客户那边已经开始阴阳怪气。

「这周，给我弄到 90% 以上。」

会议室安静了两秒。然后产品经理小满举了手——她其实半个工程师，平时最爱折腾新工具。

「这个简单，」她说，「现在不是有那种**自动优化 prompt** 的工具吗？我把咱们的提示词喂进去，让它自己跑一晚上，明早不就改好了。让 AI 改 AI 的作业，多省事。」

坐在角落、四十五岁、做过十年搜索推荐的老郭，从保温杯后面抬起眼皮，慢悠悠说了一句:

「行啊。那你先回答我一个问题——**你怎么知道它改完是真变好了，还是看起来变好了？**」

小满没接上话。

这就是故事的开始。后面整整一个季度，他们用一连串翻车，把这个问题拆成了十二小问。

## 第一章 体检：这活儿到底值不值得优化？

小满还是按自己的想法跑了。她找了个开源优化器，把小蜜的 prompt 丢进去，设定「跑 6 轮，每轮拿 30 条工单打分，挑分最高的」。

第二天早上，她兴冲冲地汇报：

「跑出来了！新 prompt 又长又专业，**开发集准确率 82% → 87%**，涨了 5 个点！」

老郭没鼓掌。他做了件扫兴的事：把那个 87% 的新 prompt，拿去跑了一批**没参与过优化的 100 条新工单**。

屏幕上的数字是 **79%**。

比最开始那个朴素的 82% 还低。

「那涨的 5 个点，」老郭说，「是从 30 条小样本的噪声里捡来的。优化器没把任务学好，它把这 30 条题的答案背下来了。」

小满有点懵：「自动优化……还能越优化越差？」

老郭转过电脑，给她看一篇 2026 年的论文（代号 Coin Flip，抛硬币）。研究者跑了 **72 次**自动优化，结果让人脊背发凉:

> **49%** —— 将近一半的优化结果，比啥都不优化的 zero-shot 还差。跟抛硬币一个概率。

「所以**自动优化不是稳赚**，」老郭说，「它有差不多一半的可能，把你的 prompt 改烂。你昨晚那一跑，运气好踩在差的那一半，还以为赚了。」

他在白板上写下第一条规矩:

> **洞见 1：动手之前先体检。** 别一上来就花钱跑优化器。先量两个东西——
> - **提升空间（headroom）**：最好的候选，比 zero-shot 高多少？
> - **噪声地板（noise floor）**：几个候选之间，分数本来就抖动多少？
>
> **如果提升空间没超过噪声地板，停手，别优化。** 你看到的「涨」很可能只是抖动。

小满不服气：「那总有能优化好的任务吧？」

「有，」老郭说，「但**得挑对任务**。」他指了指小蜜要干的三件事。

「『把工单总结成一句话』——这种开放写作，你怎么打分？只能再叫一个模型当评委。评委今天觉得 A 好，明天觉得 B 好，分数一变你根本不知道是 prompt 真变好了，还是评委今天口味变了。」

「但『抽成 {intent, entities, urgency} 的 JSON』不一样。漏抽了 urgency，那一列的分**当场就掉**，而且你能把失败掰开看：是漏抽、标错、格式坏、还是过度推断。**因果看得清。**」

那篇 Coin Flip 论文也印证了这点：六种优化方法里，唯一一个**全部都比 zero-shot 强**的任务，恰好是个结构化 JSON 抽取任务（提升 +6.8）。

> **洞见 2：第一批，先挑「能客观打分」的任务练手。** 信息抽取、分类、工具调用这种，失败能拆解、因果看得清。开放写作很重要，但别拿它当你的第一块试验田——你会被评委的口味带着走。

小满把小蜜的三件事重新排了序：先死磕「抽 JSON」，把「总结成一句话」往后放。

这是他们学会的第一件事:**优化之前，先证明这活儿值得优化、而且看得清效果。**

## 第二章 把失败变成证据，而不只是一个红叉

挑好了任务，小满准备再跑一次优化器。老郭拦住她:「这次先别喂分数给它。先喂**证据**。」

他随手抽了一条小蜜抽错的工单:

> 用户原话：「我要退货，并且投诉你们客服态度！」
> 小蜜抽出来的 intent：**投诉**（漏了「退款」）

「如果你只记『这条错了』，」老郭说，「你得到的是一个红叉，一个光秃秃的分数。你拿着这个红叉，根本不知道该改 prompt 的哪一句。」

「但如果你把它记成一句**诊断**——」他在便签上写:

> 「当一条工单同时包含『动作请求』和『情绪发泄』时，模型只抓情绪、漏掉动作。」

「——这句话直接告诉你该补哪条规则。这才是改 prompt 的燃料。」

老郭管这叫「把失败变成可编辑的证据」。他翻出一篇经典论文（ProTeGi）佐证：用失败样本生成一句句文字「诊断」、再据此改写，比原始 prompt 平均提升 **15.3%**，最猛的任务上能到 **+31%**。

但他话锋一转:「不过这里有个坑。」

同一篇论文里还有个对照实验：让一个 AI agent **自己反思、自己改 6 轮**，结果反而把起始 prompt 越改越差。「**文字『梯度』只是个比喻，不是真的数学梯度。**模型很会写一段听起来很有道理的反思，然后顺着这段错误的反思，越改越自信地跑偏。」

实习生阿哲凑过来问了个「蠢问题」：「那怎么防止它一开始就猜错方向？」

老郭说这问题一点都不蠢——这正是另一篇 2026 论文（VISTA）发现的「反思的阴暗面」。在一组本身有缺陷的种子上，某个流行的反思式优化器从 23.81% **掉到了 13.50%**：它锁定了一个错误的根因，然后在错误的方向上奋勇前进。研究者把「**猜根因**」和「**改写**」这两步硬拆开之后，分数才恢复到 **87.57%**。

「所以，」老郭把规矩写上白板:

> **洞见 3：把每个失败样本记成证据，而不只是一个分数。** 每条失败都记下：模型答了啥、正确答案、错误类型、一句具体诊断。先把 20 条失败整理成一张「失败类型表」，再让优化器**只基于这张表**改。
>
> **洞见 4：先列「失败根因假设」，再改 prompt。** 别让模型一边猜原因一边改。对同一批失败，强制列出 2–3 个**互斥**的假设，比如:
> - A：缺判定标准 → 候选 A：补一条 urgency 的判定规则
> - B：缺边界示例 → 候选 B：加两个「看似急其实不急」的例子
> - C：中性词误判 → 候选 C：声明「尽快/马上」不单独算紧急
>
> 三个候选分别去验证集上打分，**谁真把错误率拉回来用谁。**

阿哲后来把这套流程画成一句口诀贴在工位上:**「分数告诉你坏了，诊断才告诉你哪儿坏了。」**

## 第三章 别被模型骗：怎么「选」和怎么防「越改越长」

有了证据和假设，优化器这次生成了五个候选 prompt。小满的本能是:挑开发集分数最高的那个。

老郭又拦住她:「**别只看一个数。**」他让阿哲把五个候选摊成一张账:

```
候选   开发集分   格式错   长度    选不选
C1     0.86      0%      1.0x    ✔ 留
C2     0.88      6%      1.2x    ✗ 弃（分最高，但格式错升高了）
C3     0.87      0%      2.4x    ✗ 弃（太长，像在过拟合）
C5     0.86      0%      1.0x    ✔ 留
```

「分最高的是 C2，但它格式错误率从 0 涨到了 6%——这种 prompt 一上线，下游解析 JSON 就开始报错。**只看平均分，会把这种代价藏起来。**」

他们最后没用 C2，而是综合「分数 + 格式 + 长度」三个维度挑。而且老郭定了条铁律:**那个历史最佳（best-seen）永远单独存着，不许被新候选直接覆盖**——万一新的是假提升，随时能滚回去。

这套「多生几个、用验证集选、别让模型自评『我这版最好』」的做法，论文里也有数:经典的 ProTeGi 用 beam 搜索来选候选，在好几个任务上明显赢过「贪心地只留当前最好」。

「再说说 C3 为什么被弃，」老郭指着那个 **2.4 倍长**的候选，「**prompt 变长、变复杂，十有八九不是进步，是过拟合。**」

他给小满看了那个被弃候选里优化器自己加的几条规则:

```
- 若提到「发票」且含数字，intent 一定是退款      ← 这是某一条样本的补丁
- 若出现「经理 / 投诉」，urgency 一律 high        ← 把局部特例当成全局规则
- 感叹号超过两个 → 倾向 high                      ← 在拟合标注员的打字习惯
```

「每条看着都挺『专业』，其实都是在给训练样本打补丁。开发集 +5，一上压力测试集 −8。」

这事有论文撑腰，而且数字很吓人。一篇 2026 论文（PrefPO）发现：如果放任不管，一个流行的优化器能把 prompt 撑到 **14.7 倍长**，而且 **86%** 的改动其实是在**偷偷篡改任务本身**（比如把「严格抽取」悄悄改成「尽量抽取」来骗分）。另一篇（TextReg）加了个「别乱膨胀」的正则化约束，结果**又短又更准**，反而比放任版高出近 10 分。

> **洞见 5：不要用第一版，多生几个再用验证集选。** 怎么「选」和怎么「生成」一样重要。每轮都记下:种子、所有候选、各自分数、为什么留 / 为什么弃、历史最佳、回滚点。把选择权交给验证集，别交给模型的自我感觉。
>
> **洞见 6：prompt 越改越长，往往是过拟合，不是进步。** 候选如果变长了，必须说清楚每一条新规则**对应哪几个失败样本、而且扛得住压力测试**，否则一律拒绝。标准是「必要、可追溯、不过拟合」，不是「越短越好」。

## 第四章 到底该拧哪个螺丝？

抽 JSON 的准确率爬到了 86%。再往上，卡住了。

小满一遍遍地重写那段 instruction，措辞改了又改，分数纹丝不动。她有点崩溃:「我都把指令打磨成艺术品了！」

老郭说了句让她愣住的话:「**你一直在拧同一个螺丝。可这台机器上有好几个螺丝。**」

他在白板上把小蜜拆成了一张「零件清单」:

```
任务指令 task_prompt        v3   [可改]
示例     examples           v2   [可改]
推理模式 prompting_pattern  零样本 [可改]  ← 真 bug 常藏在这
工具说明 tool_schema        v1   [可改]
评委     evaluator          v1   [冻结]  ← 评委不许动（防作弊）
安全规则 safety_rules       v1   [冻结]  ← 安全/权限不进搜索空间
```

**第一个被忽略的螺丝是「示例」。** 老郭让小满做个对照:instruction 一个字不改，只精心换三个示例——专挑那些容易错的边界:

```
例1（动作+情绪并存）：要退货还要投诉  → intent:[退款, 投诉]
例2（看似急其实不急）：方便时帮我看下  → urgency: low
例3（多实体）：订单 123 和 456 都改地址 → entities:[123, 456]
```

准确率一下动了。这事论文里特别明确:有篇研究（Teach Better）发现，**instruction 完全不改、只优化示例**，分数就能涨 12.63 分；而且换个模型上，「优化示例」(75.77) 明显赢过「只优化指令」(65.91)——**精选 3 个示例，甚至比把所有示例都塞进去还好。**

「大家的本能都是去改 instruction 的措辞，」老郭说，「却忘了那个常常更有效的变量——**给模型看哪几个例子。**」

**第二个被忽略的螺丝是「推理模式」。** 阿哲负责的一个子任务（判断工单陈述是否自相矛盾）准确率低得离谱。老郭看了一眼:「你用的是零样本直接问。这个任务得让它**一步步推理**再答。」换了推理模式，没改一个字的措辞。

这个的论文数字最夸张——有篇研究（AutoPDL）发现，光是**把推理模式换对**（不是重写措辞），某个任务从 **6.5% 飙到 74.0%**，涨了 67.5 个百分点。

**第三个被忽略的螺丝是「工具说明」。** 小蜜要调一个「发起退款」的工具，老是把金额传错。小满在 agent 指令里写「请正确填写金额」，写了八百遍也没用，模型还是传 `amount: 99.5`。

老郭看了一眼工具的定义，改了一行注释:

```
amount: integer   // 单位是「分」，99 元应填 9900，不接受小数
```

问题没了。「**模型不是不听话，是工具没把单位说清楚。**你改 agent 的 system prompt 改到天荒地老都没用，因为 bug 根本不在那个螺丝上。」论文（JTPRO）里，把「全局指令」和「每个工具的说明」一起优化，整体成功率从 51.27% 提到了 64.46%。

> **洞见 7：有开发集时，「换给模型看的例子」常比「改指令措辞」更有效。** 至少比一比:不给例子 / 随便给例子 / 精选例子 / 指令也优化。别绞尽脑汁只盯着 instruction。
>
> **洞见 8：先标清楚你改的是哪个「零件」。** 每次优化都列一张零件清单，标好哪些可改、哪些冻结。**评委、测试集、安全规则必须冻结**——这是防止模型「作弊刷分」的底线。
>
> **洞见 9：工具调错时，要改的常是「工具说明」，不是 agent 指令。** 把工具评测拆成「选得对不对 / 参数填得对不对 / 整体成不成功」三个分开记，才知道瓶颈在哪一层。

这一章他们学到的，其实是同一件事:**「优化 prompt」这句话太笼统了。真实系统里，失败可能来自示例、推理模式、工具说明、甚至评委——先标清你拧的是哪个螺丝，否则你连「改好了没」都说不清。**

## 第五章 记忆与分身：更多 ≠ 更好

小蜜准确率上了 91%，Diana 终于笑了。但产品要往「越用越聪明」走，新需求来了:给小蜜加**记忆**，让它记住处理过的老工单；再拆成**多个 agent** 分工协作。

小满这次学乖了，但还是踩了两个新坑。

**第一个坑：记忆什么都记。** 她让小蜜把所有历史对话一股脑塞进上下文，想着「记得越多越聪明」。结果小蜜把 A 产品线的退款规则，错套到了 B 产品线的工单上——A 线七天无理由，B 线定制款不退，它记混了。

「**没过滤的记忆不是资产，是污染源。**」老郭说，「而且更贵——你每次都把一大坨历史塞进去，token 烧得哗哗的。」

正确的做法是给每条记忆都贴上标签:这条是从哪个任务来的、适用条件是什么、质量打几分、什么时候该忘掉。检索时**只取 3–5 条**已经验证过、且适用条件匹配当前工单的。

论文数字很有说服力:一篇研究（MemAPO）用「双记忆」机制，**又准又便宜**——70.7% vs 63.6%，成本 \$0.31 vs \$0.70。另一篇（ERM）发现，**过滤过的记忆收敛快了近一倍**（第 7 步就到 68.6，原始记忆磨到第 13 步才 58.5），而不加筛选的原始记忆**基本没带来收益**。

**第二个坑：多个 agent 出错就整体重写。** 小蜜拆成了「抽取 → 补全 → 回复」三个 agent。一旦最终回复错了，小满的本能是把三个 prompt 全推翻重写。

老郭让她先看 trace（执行轨迹）:

```
抽取 agent   局部对不对？  ✗  ← urgency 在这一步就丢了
补全 agent   局部对不对？  ✓  （但它是基于错误的输入在补，局部对、全局错）
回复 agent   局部对不对？  ✓
```

「根因在抽取那一步。补全和回复两个 agent **自己都没做错**，它们只是接到了错的输入。你要是把三个一起重写，等于砍了两个无辜的，还可能引入新 bug。」

这在多 agent 领域有个正经名字叫 **credit assignment**（责任归属）。论文（Temporal/Structural Credit）实测:**只更新真正出错的那一步**，分数从 55.13 涨到 64.63；而**无差别地更新所有 agent，反而掉分**。

> **洞见 10：只有「过滤后的记忆」才有用。** 给每条记忆贴元数据（来源任务、适用条件、质量分、遗忘条件），只检索已验证、且匹配当前任务的 3–5 条。别把历史对话整坨塞进去。
>
> **洞见 11：多 agent 出错，先找「是谁的责任」，别整个重写。** 在 trace 里记下每个 agent / 每一轮的「局部对不对、对后续有没有帮助、最终成没成」，找到**第一个引入错误**的那一步，只改那一块。

这一章的教训，其实和第四章是同一种智慧:**先定位、再动手。无论是拧螺丝还是抓责任人，「精准」永远打败「全部重来」。**

## 第六章 听来的，是线索，不是证据

故事快结束时，发生了一段小插曲。

某天小满刷到一条转发上万的爆款帖:

> **「GEPA 用少 35 倍的算力，吊打强化学习 RL！自进化时代来了！」**

她激动地冲进会议室:「郭工！咱们的优化器是不是该全换成 GEPA？人家能替代 RL 了！」

老郭没急着否定，他做了件事——**把那篇被转疯了的原论文翻出来读了一遍。**

读完他说:「GEPA 是个**反思式的 prompt 进化方法**——靠读执行轨迹、做 Pareto 选择来改 prompt。它**压根就不是一个 RL 系统**。『替代 RL』这个说法，是传播过程里为了好转发，把边界和前提全删了之后憋出来的一句口号。」

「热度只能说明它**被传得广**，」老郭把最后一条规矩写上白板，「**不能说明它有效，更不能说明它能替代什么。**」

> **洞见 12：社媒和二手文章是「线索」，不是「证据」。** 读到一个让你心动的技巧/结论，先逼自己抽六个字段:**用的什么数据集 / 什么指标 / 跟谁比的 baseline / 什么模型 / 多少成本 / 有哪些失败案例。** 缺任何一个，就把它降级成「线索」——进你的结论之前，**必须回去追到一手论文、官方文档或代码**。
>
> 例外:那种自带代码、数据、可复现实验的高质量工程博客，可以升级为更强的证据——**看细节，不看平台。**

小满后来养成个习惯:每次看到「X 吊打 Y」的标题，先在心里默念那六个字段。十有八九，标题里一个都没说。

## 尾声：老郭白板上最后留下的三句话

那个季度结束，小蜜稳在了 92%，而且——更重要的是——**他们能说清每一分是怎么来的，也能在任何一步滚回去。**

收工那天，老郭把白板上密密麻麻的十二条擦掉，只留下三句话。他说，要是哪天忘了前面所有细节，记住这三句就够了:

> **一、动手之前先问「值不值得」。** 提升空间没超过噪声地板，就别花那个钱。
>
> **二、改 prompt 的本质，是「证据 → 假设 → 多候选 → 验证集筛选 → 可回滚」，不是让模型润色一遍。**
>
> **三、更长、更多、更复杂，都不是默认的好。** 先证明它没带来过拟合、没带来污染、没让你说不清因果。

小满看着这三句话，忽然想起季度初自己那句「让 AI 改 AI 的作业，多省事」。

她现在知道答案了:让模型自己改 prompt，技术上当然可以。但**「改」很容易，「知道改对了没有」才是真功夫**——而后者，恰恰是模型替你包办不了的那部分。

那不是一个能让你睡个好觉的优化器。那是一套工程纪律。

—— 完 ——
