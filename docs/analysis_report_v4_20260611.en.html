⓪ Executive summary

The report's main judgments in one page. Each can be traced to an evidence chain in the body; see the masthead notes for evidence levels.

Overall judgment

As of 2026-06, the field of automatic prompt optimization (APO) has converged from "find a smarter rewriting algorithm" to "build an auditable optimization engineering practice": the real classification axis on the method side is the information density of the feedback signal (scalar → critique → full trace), and the entry ticket on the engineering side is the five-piece kit (prompt + dataset + metric + trace + constraints). Algorithm and infrastructure are both indispensable — papers provide the mechanism and the numbers, while running in production relies on versioning, traces, validation-set isolation, and rollback.

The five most important actionable judgments

  1. Do a checkup before optimizing. Under some settings, 49% of optimization runs are worse than zero-shot (Coin Flip, 72 runs, Haiku 4.5); the only task that yields positive returns for all 6 methods is the structured JSON output task. If the headroom doesn't exceed the noise floor, don't spend the money. A
  2. With a dev set, optimize examples first, not instructions. MIPROv2's triage conclusion: for most tasks, optimizing bootstrapped demonstrations works better than optimizing the instruction, and joint optimization is best in 5 of 7 tasks; instruction optimization is decisive only on tasks "with hidden conditional rules." A
  3. Feedback information density determines sample efficiency. From scalar (APE/OPRO) to failure critique (ProTeGi/TextGrad) to full execution trace (GEPA), the information extracted per rollout keeps rising — GEPA used about 3,936 rollouts to surpass GRPO (RL) which used 24,000 rollouts. Choose a method by looking first at what signal your system can produce. A
  4. Zero-cost structural transformation is the new control baseline. Repeating the entire prompt verbatim once wins significantly in 47/70 cases across 7 models in non-reasoning mode, with 0 significant losses — any search-based optimization gain should first answer "can it beat this zero-intelligence transformation?" Arecent
  5. No five-piece kit, no optimization loop. The optimizer's input is prompt + dataset + metric + trace + constraints, not a piece of text to be polished; without an eval, so-called optimization is just stylistic rewriting. This is the same conclusion that four engineering channels converged on independently. B

Biggest risks

Prompt-bloat-style overfitting (left unchecked, TextGrad bloats the prompt to 14.7× length, with 86% covertly altering the task definition), reward hacking (the optimizer reaches the evaluator), memory contamination (unfiltered history causes cross-task negative transfer), vendor number hallucination (task/metric/presentation biased toward the narrative), and cross-model fragility — the same method configuration that wins across the board on Haiku falls below zero-shot in 14 of 24 means when moved to Nova Lite.

Selection guide: answer six questions before choosing a method

  1. Do you have a held-out set and a scorer? No → don't choose a method yet, go assemble the five-piece kit (④ Engineering implementation); without an eval, any "optimization" is just stylistic rewriting.
  2. Did you pass the headroom checkup? Run the checkup with HM-01 (insight 01); if headroom doesn't exceed the noise floor → stop. While you're at it, first test the zero-cost structural transformation (insight 13) as a control baseline.
  3. Do you have a dev set? First run the exemplar optimization baseline (HM-03): for most tasks it's more effective than changing the instruction (MIPROv2 / Teach Better).
  4. Does the task have "hidden conditional rules that a few examples can't express"? Yes → only then is instruction optimization worth doing (MIPROv2 triage conclusion); otherwise examples + light instruction tweaks are enough.
  5. Can your system produce an interpretable trace (agent / tools / multi-module)? Yes → reflection-style methods (GEPA family) have an information-density advantage; if you can only give a scalar score → the advantage narrows, so honestly use APE/OPRO-style search.
  6. Can the complex method beat the floor baseline? Any new method's control group must include at least APE (propose-then-select) and OPRO (scalar trajectory); if it can't beat them, it isn't worth introducing.

③ Workflow insights Part A (14-item backbone · numbering aligns with unified catalog I-01..12, with 13/14 new in this edition)

Before you act: is this task worth optimizing?

The most easily skipped — and most money-saving — step: before spending money running an optimizer, first confirm whether there is room to improve, and whether the task type lets you see the effect clearly.

Insight 01Evidence A

Confirm there is room to improve before paying to run an optimizer

Automatic optimization is not a sure win — under some settings it has nearly a one-in-two chance of making the prompt worse than zero-shot.

An example (illustrative)

Ticket three-way classification, first prompt:
"Classify the ticket into: complaint/inquiry/refund"
Tested on 30 samples: 82% accuracy
Optimizer runs 6 rounds → a longer, "professional" new prompt
dev: 87% ↑ (looks like it went up)
100-sample test set: 79% ↓ lower than the original
those 5 points were picked from small-sample noise

What the papers say

49%Coin Flip: 49% of 72 optimization runs are below zero-shot (Haiku 4.5)
~3 stepsProTeGi: peaks in about 3 steps, after which it mostly overfits the training set

Source: coin-flip-2026 (recent-preprint), protegi-2023; for the controlled counterexample that "adding a risk term to the selection signal can't save it," see prose-2026

What you should do

  1. Run zero-shot + a hand-written baseline, recording a score for each.
  2. Generate 10–20 candidates and score them with the same batch of 20 held-out samples.
  3. Look at headroom (best candidate − zero-shot) and noise floor (jitter between candidates).
  4. If headroom doesn't exceed the noise floor → stop, don't optimize.
Cross-channel corroboration: web_search's WHM-01 "Optimizer Intake Checklist" and the Ask HN "AI evals half-baked" pain-point signal, Twitter tw-insight-01 "have a test set and scorer first," and Zhihu ZHI-01 "shift from tricks to eval-driven" all point independently to the same upfront discipline.
Boundary: tasks whose combinatorial structure can only be discovered through multi-round search will be underestimated by this checkup; the threshold must be recalibrated by sample size and scorer.
Insight 02Evidence A/B

For the first batch, pick tasks that "can be objectively scored"

Task type determines whether you can see clearly whether optimization helped. With open-ended writing, even after optimizing you can't tell whether it truly got better or the judge's taste changed.

An example (illustrative)

Task A: summarize a ticket into one sentence
Can only rely on another model as a judge
The judge's taste changes daily → score changes can't be attributed
Task B: extract into {intent, entities, urgency} JSON
Missed urgency → that field's F1 column drops directly
Failures break down into missed/mislabeled/bad-format/over-inferred

What the papers say

+6.8Coin Flip: the only task where all 6 methods beat zero-shot is the structured JSON task HelpSteer2
+16% → ~1%KG extraction: triple F1 +16% on the same dataset, only about 1% when the dataset changes

Source: coin-flip-2026, apo-kg-construction-2025

What you should do

  • For the first validation batch pick: information extraction (JSON), text classification, tool calling.
  • Don't use open-ended generation as your first testbed for now — causality is hardest to see clearly.
Cross-channel corroboration: maps to unified catalog I-11; Twitter conclusion-01/02 list "has a held-out set, classifiable failures, an interpretable intermediate process" as preconditions for a method to hold; web_search WPI also stresses that subjective writing tasks must first define a rubric, otherwise the metric is unstable.
Boundary: open-ended generation matters greatly, it's just not suitable as the "causally clear" first validation batch.
③ Workflow insights Part B

How to learn something from failure

A score tells you "it broke"; a textual diagnosis and root-cause hypothesis tell you "where to fix." This is the real fuel of prompt improvement. Terminology reminder: the so-called "textual gradient" is only an engineering metaphor, not a mathematical gradient — this report uniformly understands it as "feedback-driven rewriting."

Insight 03Evidence AEvidence upgrade

Turn failure samples into "editable evidence," not just a score

That accuracy number gives you nothing to grab onto; a specific failure diagnosis points directly at what the prompt should add.

An example (illustrative)

Ticket: "I want to return the item and complain about service"
Model extracts intent: complaint (missed refund)

Record only the score → this one = wrong (a red X, no lead)
Record as evidence → "When a ticket contains both an 'action request'
            and 'emotion,' the model only catches the emotion and
            misses the action" ← tells you directly which rule to add

What the papers say

+15.3% / up to +31%ProTeGi: generating textual feedback from failure samples then rewriting, versus the original prompt
0.760 vs 0.110SPEAR: giving the optimizer Python tools for error analysis, kappa on hard tasks far exceeds TextGrad

Source: protegi-2023, spear-2026 (recent-preprint: error analysis can use code tools rather than relying solely on long context), textual-gradients-flawed-metaphor-2025 ("gradient" is a metaphor, not a real gradient)

What you should do

  • For each failure case record: prediction, gold, error_type, critique, candidate_id.
  • First organize 20 failures into a "failure type table," then let the optimizer change only based on this table.
  • When failure samples are many (>100), consider running the analysis in code (pandas/groupby to aggregate error types) rather than stuffing everything into context — SPEAR shows long-context aggregation is unreliable.
Cross-channel corroboration: GitHub's compare-first rewrite (linshenkx/prompt-optimizer source) preserves failure evidence and diffs; Twitter tw-insight-02 "let the optimizer read the failure process, not just the final score"; web_search WHM-03 and Zhihu HM-ZH-02 both require layered diagnosis by failure owner.
Boundary: critique carries judge bias and must be bound to samples and metrics, never treated as true causality. The Flawed-Metaphor warning: textual feedback gains often come from formatting and meta-instructions, even prevalence hacking — treat critique as a lead, not as truth.
Insight 04Evidence Arecent-preprint

List "failure root-cause hypotheses" first, then change the prompt

The most common way reflection-style optimizers crash is guessing the wrong direction at the start, then growing more confident while editing in that wrong direction.

An example (illustrative)

urgency misjudged 30% of the time. Don't directly "let the model fix it"; list hypotheses first:
 A missing decision criteria  → candidate A: add urgency decision rules
 B missing boundary examples  → candidate B: add 2 "looks urgent but isn't" examples
 C neutral-word misjudgment   → candidate C: state that "ASAP/right away" alone doesn't constitute urgency
All three candidates validated on a minibatch; use whichever actually pulls the 30% back

What the papers say

23.81% → 13.50% → 87.57%VISTA: GEPA actually drops to 13.50% on a flawed seed; after decoupling "hypothesis generation/rewriting" it recovers to 87.57%

Source: vista-reflection-dark-2026, llm-prompt-optimizers-2024

What you should do

  • For the same batch of failures, force the generation of 2–3 mutually exclusive root-cause hypotheses (missing rule / missing example / format conflict).
  • Generate one candidate per hypothesis and score each separately; the run record stores failure_hypothesis, not just critique.
Cross-channel corroboration: this item is currently supported mainly by a single layer, arXiv (VISTA), with no independent evidence from engineering channels yet — it belongs to the "mechanism clear, awaiting engineering and this project's validation" category and is listed under P1.
Boundary: open-ended writing tasks have fuzzy root causes, and multi-hypothesis validation may cost more than it returns.
③ Workflow insights Part C

Don't let the model fool you: candidate selection and anti-bloat

LLM candidates have extreme variance. How you "select" and how you prevent "getting longer and longer" are the two gates that keep pseudo-improvements out.

Insight 05Evidence A

Don't adopt the first version; generate several and select with a validation set

"How you select candidates" is as important as "how you generate candidates." Hand the selection power to the validation set, don't let the model self-assess "this is my best version."

An example (illustrative): a candidate ledger

Cand  dev-score  format-err  length  select?
C1   0.86   0%    1.0x   ✔candidate
C2   0.88   6%    1.2x   ✗reject (format errors rose)
C3   0.87   0%    2.4x   ✗reject (too long, suspected overfitting)
C5   0.86   0%    1.0x   ✔candidate
Don't just pick the highest dev C2, but select across three dimensions; best-seen is never directly overwritten

What the papers say

0.85 / 0.67 / 0.88ProTeGi: beam selection clearly beats greedy on Jailbreak/Liar/Sarcasm; bandit > uniform selection
+12.44 vs +6.05GEPA ablation: the Pareto candidate pool itself is the main contribution, clearly beating greedy selection

Source: protegi-2023, gepa-2026, promptbreeder-2023

What you should do

  • Each round store seed/candidates/scores/selection_policy/rejected_reason/best_seen/rollback_point.
  • Generate 5 candidates and select Pareto candidates across three dimensions: dev score + format errors + length.
Cross-channel corroboration: GitHub insight cards list the candidate ledger as a reusable structure (→ HM-04); Twitter method-01 "generate and select candidates under metric+trace constraints" contrasts scalar-only with trace-aware feedback — the industry version of the same principle.
Boundary: the larger the search budget, the easier it is to overfit dev, so validation/test layering must be there as a backstop.
Insight 06Evidence AEvidence upgrade

Prompt hygiene is a proactive optimization metric: getting longer and more complex is often overfitting, not progress

Longer after optimization, more rules, looking more professional — very likely patching the training samples. Hygiene shouldn't only be an after-the-fact check; it should enter the selection function.

An example (illustrative): "pseudo-rules" the optimizer adds

- If "invoice" is mentioned and a number appears, intent is always refund   ← a patch for one sample
- If "manager/complaint" appears, urgency is always high      ← a local edge case as a global rule
- More than two exclamation marks → lean high                 ← fitting annotator habits
dev +5, stress test set −8

What the papers say

14.7x / 86%PrefPO: left unchecked, TextGrad bloats to 14.7× length, 86% covertly altering the task itself (PrefPO only 37%)
+10.0 / +9.9TextReg: adding length+scope regularization to control bloat actually scores higher than TextGrad

Source: prefpo-2026, textreg-2026, causal-edit-level-2026 (all recent-preprint; the last shows meta-instructions and complexification are negatively associated with math/multi-hop tasks — different tasks respond to edit directions heterogeneously)

What you should do

  • Write hygiene into candidate selection: length_ratio, repetition, edit_family, dev_test_gap, OOD/stress_delta enter the Pareto together with the dev score.
  • If a candidate gets longer, it must explain which failure samples each new rule is bound to and pass the stress test, otherwise reject it.
  • Overfitting is not only the dev-test gap: rules narrowing (scope shrinkage) and representation bloat are equally signals (TextReg's formalization).
Cross-channel corroboration: the anti-pattern tables of GitHub and web_search both single out "only watching the average rise" (which can mask schema drift, minority-class regression, safety degradation, cost bloat); web_search also lists "trusting vendor improvement of X%" as a same-source risk. From a survey perspective, this thread is the empirical version of "constrained optimization."
Boundary: some complex tasks genuinely need clearer constraints — the standard is "necessary, traceable, not overfitting," not "short."
③ Workflow insights Part D

What exactly to change (not just that prompt)

In real systems failure may come from examples, the prompting pattern, or the tool description — label clearly which component you are changing first, otherwise you can't attribute.

Insight 07Evidence AEvidence upgrade

With a dev set, choosing "the examples shown to the model" is often more effective than changing the instruction

People instinctively change the instruction wording, yet overlook the often-more-effective variable — which examples to show the model. This item has been upgraded from single-paper support to multi-paper independent support.

An example (illustrative)

Leave the instruction unchanged, only swap 3 hand-picked examples (covering error-prone boundaries):
 Ex1 action+emotion coexist: want to return and also complain → intent:[refund,complaint]
 Ex2 looks urgent but isn't: check it for me when convenient → urgency: low
 Ex3 multi-entity: change the address for orders 123 and 456 → entities:[123,456]
Often yields more than racking your brain to rewrite the instruction

What the papers say

+12.63Teach Better: without changing the instruction, optimizing only examples reaches 72.92 (PaLM 2 / BBH)
5/7MIPROv2: joint optimization is best in 5 of 7 tasks; triage conclusion — for most tasks demo optimization > instruction optimization

Beyond search: zero-cost transformation and "the optimizer itself"

v3's 12 insights were all built around the "search/optimization loop." This round of deep-dives across channels exposed two blind spots outside the loop: one is a deterministic structural transformation that uses no search (cheaper than all search methods, yet often skipped as a control), and the other is the optimizer and judge themselves, treated as thin air inside the loop.

Insight 13Evidence Arecent-preprintnew in this edition

Before paying to search, first try the zero-cost deterministic structural transformation

A fixed transformation that changes no semantics — repeating the entire prompt verbatim — wins 47 and loses 0 across 7 models in non-reasoning mode; while search-based optimization has nearly a one-in-two chance of being worse than zero-shot.

An example (real task, paper numbers)

Transformation: <QUERY>  →  <QUERY><QUERY> (verbatim repeat, no wording change)

NameIndex (pick the 25th of 50 names):
Gemini 2.0 Flash-Lite  21.33% → 97.33%
Mechanism: under causal attention, early prompt tokens can't see later ones;
after repetition, every token in the second pass can attend to the full first pass

What the papers say

47/70 wins 0 lossesNon-reasoning mode, 7 models × 10 configs, McNemar p<0.1; the Padding control is ineffective → the gain comes from the repetition itself
5/28Once step-by-step reasoning is on it's basically ineffective (reasoning models already often restate the question on their own)

Source: prompt-repetition-2025 (Google Research; doesn't increase output length or latency, the repetition happens in the parallelizable prefill stage)

What you should do

  • On non-reasoning, low-latency production paths (classification/extraction/short QA/routing), first test repetition ×2; for long-context indexing tasks try ×3.
  • Add it to the cheap baseline transformation set: any APO method's reported gain in this project should first answer "can it beat this zero-intelligence transformation?"
  • Prefer it in scenarios where information order is uncontrollable (concatenated retrieval results, form transcription) — order sensitivity is exactly what it fixes.
Cross-channel corroboration: currently single-layer arXiv (recent-preprint); the mechanism (causal attention order sensitivity) cross-validates with the section-local edit direction of Modular Prompt Optimization. The contrasting reading with Coin Flip: the searched wording is model-specific and fragile, while the deterministic structural transformation loses 0 across 7 models — "structure" may be more transferable than "wording" (the two settings differ, so this comparison is an inference).
Boundary: basically ineffective in reasoning mode; with per-token billing the input cost is ×2; repeating only a part (e.g., only the question) yields no gain; blindly doubling an ultra-long context drives up prefill cost; it was tested on an early-2025 generation of models, and newer models may have internalized the restating behavior — which is exactly what this project's P0 three-arm A/B is meant to calibrate.
Insight 14Evidence A/Bnew in this edition

The optimizer and judge themselves are also components that must be versioned and can be optimized

The "prompt you use to change prompts" and the "prompt that scores" also determine the result — don't version them and the experiment isn't reproducible; never examine them and they become a hidden ceiling.

An example (illustrative)

The run record only stored task_prompt v3
optimizer_prompt / judge_prompt have no version number
→ three months later "the same config" can't reproduce the same result
→ is the score change the doing of the task prompt or did the judge's taste change? Can't attribute

What the papers say

71.89 → 76.38SePO: pre-train and self-evolve the prompt agent's system prompt; removing self-improvement drops back to 74.94
the only self-referencePromptBreeder: the only one of the six methods that evolves the mutation prompt too (hyper-mutation)

Source: sepo-2026 (recent-preprint), promptbreeder-2023; judge versioning is web_search WPI-10 (B-level)

What you should do

  • Give optimizer_prompt, judge_prompt, and meta_prompt each a version number, and make them required fields in the experiment record.
  • The judge used for evaluation, like the test set, is frozen by default; to change it, go through an independent review and recalibration, and after the change all old scores are invalidated and rerun.
  • Want to optimize the judge/optimizer itself? Do it on the dev set, physically isolated from the evaluation pipeline.
Cross-channel corroboration: web_search WPI-10 "the judge prompt should also be versioned, evaluated, and optimized"; GitHub autoresearch's ledger binds the optimizer config to results (GHI-05); arXiv batch1 conclusion C-03 "the optimizer itself is an artifact that must be versioned." Three layers point independently to the same governance object.
Boundary: "optimizing the judge" and "frozen evaluator anti-cheating" don't conflict — what is frozen is the evaluation pipeline's judge, what is optimized is the development-time auxiliary judge; the two must be physically separated, mixing them is reward hacking (see the anti-pattern table).

Deliberate boundaries: what we choose not to do

  • soft / continuous prompt tuning (prefix/embedding level) — focusing on discrete natural-language prompts, explicitly excluded by the scoring rules.
  • multimodal (text-to-image/video, VLM prompt optimization) — both surveys list it as a direction, but this project explicitly doesn't do it.
  • full context-engineering systems (RAG architecture, retriever optimization, multi-agent communication protocols) — treated as boundary material; but "context compression" and "intelligent context assembly," being isomorphic to prompt optimization, are registered in the backlog.
  • These are scope choices, not omissions — per the taxonomy completeness check (against 3 surveys).

Frontier gaps: directions the field has but we don't yet have a slot for

  • bi-level / thought-driven prompt optimization (the reasoning-model o1/R1 era) — the only "adjacent whole block that is entirely absent": none of the 25 empirical papers from 2025/2026 in the library touch it, and only the APE survey names it conceptually (see the gap count in the 2025/2026 frontier synthesis), the top priority for the next directed arXiv search.
  • task-agnostic / online optimization (no Dval, intent drift, non-stationary) — related to the self-evolution main line, worth a directed search.
  • constrained optimization — this report's hygiene theme (insight 06) is its empirical version, with the theoretical shell yet to be filled.
  • multi-task / negative transfer — discussed jointly with memory filtering (insight 10).
  • lighter sub-mechanisms within clusters (learned scoring, token-level editing, bandit filtering, MoE routing) — known to exist, deliberately not gone into deeply, only a pointer left.
