## 方法或实践

- Prompt / context 组织：把始终注入的提示记忆、按需检索的会话记忆、可复用技能和用户建模分层处理，避免上下文膨胀。
- Eval / 指标：文章没有给出系统 eval，只提到单一加速案例和若干 issue/用户反馈。
- 日志 / 可观测性：完整会话归档到 SQLite，并通过 FTS5 检索后摘要注入上下文。
- 发布 / 回滚：技能文件是 Markdown，可通过 create/patch/edit/delete 等操作演化；文章强调 patch 的局部性。
- 成本 / 延迟：通过技能复用和程序化工具调用减少重复推理；量化证据不足。

## 案例和失败模式

文章列出的风险集中在四类：

- 记忆治理：容量上限、关键词检索不足、会话内记忆快照不实时更新。
- 成熟度：快速迭代导致接口和运行稳定性风险。
- 生态：现成技能数量不如更大社区。
- 编排：复杂多 agent 工作流能力有限。

## 对本项目的启发

- 可转化假设：把失败轨迹压缩成可复用技能/规则，比单纯检索历史对话更能提升重复任务表现。
- 最小 eval：选择 10-20 个重复性代码或资料整理任务，对比三种条件：无记忆、检索历史摘要、使用人工/自动沉淀的技能文件。
- 需要新增的数据或脚本：任务轨迹记录、失败原因标注、技能文件生成模板、技能 patch diff 记录。
- 风险：自动生成技能可能固化错误经验；需要版本、审核、回滚和验证集。
