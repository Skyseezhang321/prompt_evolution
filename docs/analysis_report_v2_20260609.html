Prompt 优化与自进化：读者向洞见报告 v2

这版报告换了写法：不再罗列「哪个渠道贡献了什么」，而是按你实际会遇到的顺序，讲 12 条洞见——每条都配一个看得懂的例子、一个带出处的真实数字、一组可照抄的步骤和一条诚实的边界。它是 读者向洞见手册 的可视化摘要。

12 条洞见 真实数字带出处 可照抄步骤 诚实标注边界
一句话总结论 自动优化 prompt 不是「让模型把 prompt 改好」，而是一套「先判断值不值得 → 把失败变成可编辑证据 → 多候选 + 验证集筛选 → 可回滚」的工程纪律。
怎么读这里的数字 「论文怎么说」里的数字 = 论文设置下成立，不等于在你的任务上成立。带「示意」的 prompt/对话是讲解虚构，非实验数据。

Part A

动手之前：这个任务值不值得优化？

最容易被跳过、也最省钱的一步：在花钱跑优化器之前，先确认有没有提升空间、以及任务类型能不能让你看清效果。

洞见 01证据 A

先确认有没有提升空间，再花钱跑优化器

自动优化不是稳赚的——某些设置下，它有近一半概率让 prompt 比 zero-shot 还差。

一个例子（示意）

工单三分类，初版 prompt：
「把工单分到：投诉/咨询/退款」
30 条样本测：准确率 82%
优化器跑 6 轮 → 又长又"专业"的新 prompt
dev：87% ↑（看起来涨了）
100 条测试集：79% ↓ 比最初还低
那 5 分是小样本噪声里捡的

论文怎么说

49%Coin Flip：72 次优化里 49% 低于 zero-shot（Haiku 4.5）
~3 步ProTeGi：约 3 步达峰，之后多在训练集过拟合

来源：coin-flip-2026（recent-preprint）、protegi-2023

你该怎么做

  1. 跑 zero-shot + 手写 baseline，各记一分。
  2. 生成 10–20 个候选，用同一批 20 条留出样本打分。
  3. 看 headroom（最高候选 − zero-shot）和 noise floor（候选间抖动）。
  4. headroom 没超过 noise floor → 停，别优化。
边界：靠多轮搜索才能发现组合结构的任务会被这个体检低估；阈值要按样本量和评分器重校准。
洞见 02证据 A/B

第一批先选「能客观打分」的任务

任务类型决定你能不能看清优化有没有用。开放写作就算优化了，你也分不清是真变好还是评委口味变了。

一个例子（示意）

任务甲：把工单总结成一句话
只能靠另一个模型当评委打分
评委口味天天变 → 分数变化无法归因
任务乙：抽成 {intent, entities, urgency} JSON
漏抽 urgency → field F1 该列直接掉
失败能拆成 漏抽/标错/格式坏/过度推断

论文怎么说

+6.8Coin Flip：唯一让 6 种方法全超 zero-shot 的是结构化 JSON 任务 HelpSteer2
+16% → ~1%KG 抽取：同数据集 triple F1 +16%，换数据集只剩约 1%

来源：coin-flip-2026、apo-kg-construction-2025

你该怎么做

  • 第一批验证选：信息抽取（JSON）、文本分类、工具调用。
  • 暂时别用开放生成当第一块试验田——因果最难看清。
边界：开放生成很重要，只是不适合作为「因果清晰」的首批验证。
Part B

怎么从失败里学到东西

分数告诉你「坏了」，文字诊断和根因假设才告诉你「该改哪里」。这是 prompt 改进真正的燃料。

洞见 03证据 A

把失败样本变成「可编辑的证据」，不只是一个分数

accuracy 那个数字无从下手；一句具体的失败诊断才直接指向 prompt 该补什么。

一个例子（示意）

工单：「我要退货并且投诉客服态度」
模型抽出 intent: 投诉（漏了 退款）

只记分数 → 这条 = 错（一个红叉，没线索）
记成证据 → "当一条工单同时含『动作请求』和『情绪』时，
            模型只抓情绪、漏掉动作" ← 直接告诉你该补哪条规则

论文怎么说

+15.3% / 最高 +31%ProTeGi：用失败样本生成文字"梯度"再改写，比原始 prompt
越改越差同论文：让 AutoGPT 自己反馈改 6 轮，反而把起始 prompt 改差

来源：protegi-2023、textual-gradients-flawed-metaphor-2025（文字"梯度"是比喻，非真梯度）

你该怎么做

  • 每个失败 case 记：prediction、gold、error_type、critique、candidate_id。
  • 先把 20 条失败整理成「失败类型表」，再让优化器只基于这张表改。
边界：critique 会带评委偏见，必须绑定样本和指标，不能当成真实因果。
洞见 04证据 Arecent-preprint

先列「失败根因假设」，再改 prompt

反思式优化器最常见的翻车，是一开始就猜错方向，然后在错误方向上越改越自信。

一个例子（示意）

urgency 判错 30%。不要直接"让模型改好"，先列假设：
 A 缺判定标准   → 候选A：补 urgency 判定规则
 B 缺边界示例   → 候选B：加 2 个"看似急其实不急"的例子
 C 中性词误判   → 候选C：声明"尽快/马上"不单独构成紧急
三个候选都在 minibatch 验证，谁真把 30% 拉回来用谁

论文怎么说

23.81% → 13.50% → 87.57%VISTA：GEPA 在缺陷种子上反而掉到 13.50%；解耦"假设生成/改写"后恢复 87.57%

来源：vista-reflection-dark-2026、llm-prompt-optimizers-2024

你该怎么做

  • 对同一批失败，强制生成 2–3 个互斥根因假设（缺规则/缺示例/格式冲突）。
  • 每个假设生成一个候选，分别评分；运行记录存 failure_hypothesis 而不只 critique。
边界：开放写作任务根因模糊，多假设验证可能成本大于收益。
Part C

别让模型骗你：候选选择与防膨胀

LLM 候选方差极大。怎么"选"和怎么防"越改越长"，是把伪提升挡在门外的两道闸。

洞见 05证据 A

不要采用第一版，多生几个再用验证集选

「怎么选候选」和「怎么生成候选」一样重要。把选择权交给验证集，别让模型自评"我这版最好"。

一个例子（示意）：一张候选账

候选  dev分  格式错  长度   选不选
C1   0.86   0%    1.0x   ✔候选
C2   0.88   6%    1.2x   ✗弃（格式错升高）
C3   0.87   0%    2.4x   ✗弃（太长，疑似过拟合）
C5   0.86   0%    1.0x   ✔候选
不是只挑 dev 最高的 C2，而是综合三维选；best-seen 永不被直接覆盖

论文怎么说

0.85 / 0.67 / 0.88ProTeGi：beam 选择在 Jailbreak/Liar/Sarcasm 明显优于贪心；bandit > 均匀选

来源：protegi-2023、gepa-2026、promptbreeder-2023

你该怎么做

  • 每轮存 seed/candidates/scores/selection_policy/rejected_reason/best_seen/rollback_point。
  • 生成 5 个候选，按 dev 分 + 格式错误 + 长度三维选 Pareto 候选。
边界：搜索预算越大越容易 dev 过拟合，必须有 validation/test 分层兜底。
洞见 06证据 A

prompt 变长、变复杂，往往是过拟合不是进步

优化后更长、规则更多、看起来更专业，很可能是在给训练样本打补丁。

一个例子（示意）：优化器加的"伪规律"

- 若提到「发票」且含数字，intent 一定是退款   ← 某条样本的补丁
- 若出现「经理/投诉」，urgency 一律 high      ← 局部特例当全局规则
- 感叹号超过两个 → 倾向 high                 ← 在拟合标注者习惯
dev +5，压力测试集 −8

论文怎么说

14.7x / 86%PrefPO：放任时 TextGrad 撑到 14.7 倍长、86% 在偷改任务本身（PrefPO 仅 37%）
+10.0 / +9.9TextReg：加正则化管住膨胀，反而比 TextGrad 更高

来源：prefpo-2026、textreg-2026、causal-edit-level-2026（均 recent-preprint）

你该怎么做

  • 候选选择记录 length_ratio、repetition、edit_family、dev_test_gap、OOD/stress_delta。
  • 候选若变长，必须说明每条新规则绑定哪些失败样本并通过压力测试，否则拒绝。
边界：有些复杂任务确实需要更明确约束——标准是「必要、可追溯、不过拟合」，不是「短」。
Part D

到底该改什么（不只是那段 prompt）

真实系统里失败可能来自示例、prompting 模式、工具描述——先标清你改的是哪个部件，否则无法归因。

洞见 07证据 A

有开发集时，选「给模型看的例子」常比改 instruction 更有效

大家本能去改 instruction 措辞，却忽略了常更有效的变量——给模型看哪几个示例。

一个例子（示意）

instruction 不动，只换 3 个精选示例（覆盖易错边界）：
 例1 动作+情绪并存：要退货还要投诉 → intent:[退款,投诉]
 例2 看似急其实不急：方便时帮我看下 → urgency: low
 例3 多实体：订单 123 和 456 都改地址 → entities:[123,456]
往往比绞尽脑汁重写 instruction 收益更大

论文怎么说

+12.63Teach Better：不改 instruction、只优化示例就达 72.92（PaLM 2 / BBH）
75.77 > 65.91Gemini 1.0 Pro：优化示例 > 只优化 instruction；且精选 3 个 > 全部示例

来源：teach-better-show-smarter-2024

你该怎么做

  • 至少比较 no-example / random-example / optimized-example / instruction+optimized。
  • 固定 instruction，从候选池搜 3 个示例看留出表现。
边界：上下文紧张、示例分布偏、示例可能泄漏答案时，会伤害泛化。
洞见 08证据 A/B

先标清楚你改的是哪个「部件」

失败可能来自示例、prompting 模式、工具 schema、agent 角色、记忆或评委，不只是 system prompt。

一个例子（示意）：artifact 清单

task_prompt        v3   [mutable]
examples           v2   [mutable]
prompting_pattern  Zero-Shot [mutable]  ← 真 bug 常在这：该用 CoT/ReAct 却用 Zero-Shot
tool_schema        v1   [mutable]
evaluator          v1   [FROZEN]  ← 评委不许改（防作弊）
safety_rules       v1   [FROZEN]  ← 安全/权限不进搜索空间

论文怎么说

6.5% → 74.0%AutoPDL：只把 prompting 模式换对（非重写措辞），FEVER 跳 +67.5pp

来源：autopdl-2025、jtpro-2026、prompt-codebooks-2026

你该怎么做

  • 每次优化生成 artifact manifest，拆出各部件并标 mutable/frozen。
  • 评委、测试集、安全规则必须 frozen——防 reward hacking 的底线。
边界：简单一次性任务不需要完整 manifest，研究/生产级才需要。
洞见 09证据 Arecent-preprint

工具调错时，改的常是「工具说明」，不是 agent 指令

失败常来自工具描述和参数语义本身；改 agent 的 system prompt 再多遍也没用。

一个例子（示意）

改 agent 指令（无效）：
"请正确填写金额"
→ 模型仍不知道单位，传 amount: 99.5
改工具 schema（有效）：
amount: integer
// 单位「分」，99 元应填 9900，不接受小数

论文怎么说

51.27% → 64.46%JTPRO：联合优化全局指令 + 每工具 schema，o3-mini 整体成功率（ToolACE 1000 工具）
选对≠成功ETID：工具选得准但成功率低，瓶颈是 slot-filling 不是选择

来源：jtpro-2026

你该怎么做

  • 工具 eval 拆成 Tool Selection / Slot Filling / Overall Success 三个指标分别记。
  • 分别测「只改全局指令 / 只改 schema / 联合」，看瓶颈在哪层。
边界：外部 API schema 不可改时，用 wrapper 文档或检索上下文承载局部规则。
Part E

记忆与多 agent：更多 ≠ 更好

加记忆、加 agent 听起来总是好的，但未经筛选的记忆会污染优化，整体重写会让你无法归因。

洞见 10证据 A/B

只有「过滤后的记忆」才有用

未经筛选地堆历史，会造成跨任务负迁移，还更贵。

一个例子（示意）

无界记忆：把所有历史对话塞进上下文
→ 把 A 产品线的退款规则错套到 B 产品线工单
过滤记忆：每条带元数据
 content / source_task / applicability
 quality_score / retrieval_reason / forget_after
只取 3–5 条已验证、且适用条件匹配的

论文怎么说

70.7% vs 63.6%｜$0.31 vs $0.70MemAPO：双记忆比 TextGrad 更好且更便宜
第7步 68.6 vs 第13步 58.5ERM：过滤记忆收敛近两倍速；原始记忆无收益

来源：memapo-2026（recent-preprint）、erm-memory-2024

你该怎么做

  • 记忆 schema 含 success_template/error_pattern、source_task、applicability、quality、forget_reason。
  • 别塞历史对话，只检索已验证、匹配当前任务的 3–5 条。
边界：非平稳/安全敏感任务里，遗忘策略和 opt-out 比「记得多」更重要。
洞见 11证据 Arecent-preprint

多 agent 出错，先找「是谁的责任」，别整个重写

核心难题不是加更多 agent，而是 credit assignment——一个 agent 局部完全正确，却可能让整体失败。

一个例子（示意）

抽取 → 补全 → 回复，最终回复错了。看 trace：
 agent_1 抽取  local_valid ✗  ← urgency 在这里就丢了
 agent_2 补全  local_valid ✓  （基于错误输入 = local_pass_global_fail）
 agent_3 回复  local_valid ✓
根因在 agent_1，只改 agent_1，别动另两个

论文怎么说

55.13 → 64.63Temporal/Structural Credit：credit-guided 在 MedMCQA（+9.50）；无差别更新所有轮反而掉分
70.39 vs 65.31MASPO：联合优化平均优于顺序 baseline

来源：temporal-structural-credit-mas-2026、maspo-2026、mapro-2025

你该怎么做

  • trace 存 role_id/round_id/local_validity/successor_utility/global_outcome。
  • 标出首次引入错误的 role/round，只改那一块；耦合弱时先做 coupling test。
边界：agent 交互很弱时，联合优化成本可能不划算——先测耦合再决定。
Part F

怎么对待「听来的」方法

热度只说明传播，不说明有效。二手传播会放大结论、丢掉边界和版本。

洞见 12证据 B

社媒和二手文章是「线索」，不是「证据」

一个方法被反复转发，只能说明它热，不能说明它有效。

一个例子（示意）

爆款：「GEPA 用少 35 倍 rollout 吊打 RL！」
当证据 → 直接信「GEPA 能替代 RL」
当线索 → 翻原论文发现：
GEPA 是反思式 prompt 进化（执行轨迹 + Pareto 选择），
根本不是 RL 系统；"替代 RL"是传播简化出的口号

你该怎么做

  • 读到技巧先抽六字段：dataset / metric / baseline / model / cost / failure_cases。
  • 缺任一字段就降级为「线索」，进结论前必须追溯论文/官方文档/代码/本项目实验。

来源：twitter_web_analysis_20260608

边界：自带代码、数据、可复现实验的高质量工程博客可升级为更强证据——看细节，不看平台。

把这些用起来：首批最小验证

最值得本项目先动手验证的几条——成本低、错误可解释、跨渠道支撑强。详细设计见 实验计划。

优先级验证什么对应洞见最小任务无论成败都能产出
P0优化前体检（headroom / noise floor）01、02100–300 条 JSON 抽取判断有无优化空间，校准噪声地板
P0示例优化 vs 指令优化07同一抽取任务防止把示例收益误记成指令收益
P1直接改写 vs 根因假设改写03、04失败清晰的抽取/分类判断根因假设是否改善留出表现
P1只看性能 vs 加卫生门06同一候选池判断防膨胀门是否减少伪提升
P2工具 schema 优化0920–50 个相似工具判断改 schema 是否优于只改全局指令
P2过滤记忆 vs 原始记忆10两个相近 + 一个异质任务判断记忆是否降本或带来负迁移
给读者的三条总结：① 优化前先问「值不值得」，headroom 没超过 noise floor 就别花钱；② 改 prompt 的本质是「证据 → 假设 → 多候选 → 验证集筛选 → 可回滚」，不是让模型润色；③ 更长、更多、更复杂都不是默认的好，先证明它没带来过拟合、污染或归因困难。
