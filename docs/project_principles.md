# 项目构建原则

更新时间：2026-06-08

本文件定义 `prompt_evolution` 的长期构建原则。原则借鉴 Karpathy-inspired coding agent guidelines，但已从“减少 LLM 编码错误”适配为“减少 prompt 优化研究中的错误假设、过度工程、不可复现结论和无证据改动”。

项目的核心产出优先级是：有效 insights、可信 conclusions、可复用 helpful methods、反模式和风险边界。实验、脚本和 benchmark 是服务这些产出的验证与演示手段，不是项目目标本身。

参考来源：[andrej-karpathy-skills](https://github.com/multica-ai/andrej-karpathy-skills/tree/main)

## 1. 先定义问题，不先写 prompt

Prompt 不是孤立文案，而是任务、模型、上下文、示例、约束和评分器共同作用的结果。新增或修改 prompt 前，先写清楚：

- 目标任务是什么。
- 当前假设是什么。
- 输入样本来自哪里。
- 使用什么模型和参数。
- 用什么指标判断变好或变差。
- 哪些约束不能被自动优化改写。

检查问题：如果另一个人只看记录，能否理解本轮实验为什么要做，以及怎样判断成功？

## 2. 洞见和方法优先，最小实验服务验证

新增材料或实验前，先说明它要支持哪个洞见、结论或可复用方法。先把可迁移的经验讲清楚，再选择能验证该经验的最小实验；不一开始追求完整平台、复杂 agent workflow 或通用优化框架。

- 优先选择评分明确、运行便宜、错误可解释的任务。
- 优先手写 baseline，再加入自动候选生成。
- 没有复用价值的逻辑不要提前抽象。
- 没有明确失败模式前，不扩大搜索空间。
- 实验结论必须能回填到 insight card、method playbook、反模式或风险边界中。

检查问题：当前工作是否产出了更清晰的 insight、conclusion 或 helpful method？当前实现是不是验证该洞见或方法所需的最小可运行版本？

## 3. 一次只改一个变量

Prompt 优化最容易出现的问题是多个变量一起变化，最后无法判断收益来自哪里。默认把以下变量分开改：

- prompt 文案。
- few-shot examples。
- system prompt。
- retrieval context。
- tool policy。
- 模型和采样参数。
- 数据切分。
- 评分器和评估口径。

如果必须同时修改多个变量，必须把本轮标记为多因素实验，并避免做单变量因果结论。

检查问题：如果指标变化，能否说明主要原因来自哪个改动？

## 4. 每个结论必须有证据

项目文档中的结论需要有来源，不能只基于印象或单个输出样例。

- 论文结论要指向论文、实验设置和局限。
- 实验结论要指向 run artifact、指标、样例输出和失败案例。
- 工程判断要说明影响范围、风险和替代方案。
- 方法建议要说明适用场景、使用步骤、反例、误用风险和验证方式。
- 区分观察、推测和结论。

检查问题：读者能否沿着文档找到支撑该结论的证据？

## 5. 目标驱动执行

不要使用模糊目标，例如“优化 prompt”“提升效果”“让它更稳”。要转化成可验证目标：

```text
在 baseline X 上，
使用数据集版本 D 和评分器 E，
将主指标 Y 从 A 提升到 B，
同时格式错误率、成本或安全失败类型 Z 不恶化。
```

每个多步骤任务都应包含计划和验证方式。验证方式可以是测试、指标对比、样例检查、人工审核或文档一致性检查。

检查问题：本轮工作完成后，是否能明确判断成功、失败或需要继续实验？

## 6. 变更可追踪

Prompt 自进化如果没有版本、评估和回滚，很容易变成不可审计的行为漂移。重要变更至少记录：

- prompt 版本前后。
- 模型、参数和工具配置。
- 数据集版本和切分。
- 评分器或 judge rubric。
- 候选生成方法。
- 指标变化和成本。
- 失败模式。
- 采用或拒绝原因。
- 回滚点。

对研究追踪有意义的变更要写入 `CHANGELOG.md` 的 `Unreleased`。

检查问题：如果新版本变差，能否找到改动原因并回滚？

## 7. 精准修改

每次改动都应能直接追溯到当前目标。不要把无关重构、格式整理、术语替换或实验想法混进同一次变更。

- 匹配现有文档和代码风格。
- 只清理自己改动造成的无用内容。
- 发现无关问题可以记录或提醒，不要顺手删除。
- 修改共享工作流、评分器或数据结构时，扩大验证范围。

检查问题：每一处 diff 是否都能解释为服务当前请求或实验目标？

## 生效信号

这些原则有效时，项目应表现为：

- 实验记录能复现。
- prompt 版本能比较。
- 结论能追溯证据。
- 洞见能转化为可复用方法或反模式。
- 实验能说明验证了哪个洞见，而不是只展示分数。
- diff 干净且聚焦。
- 自动优化不会绕过安全、权限和评估约束。
- 失败案例会进入下一轮 eval 或风险记录。

## 权衡

这些原则偏向谨慎和可追踪，可能降低简单任务的速度。对于拼写修正、单行文档更新等低风险改动，可以简化流程；对于实验设计、prompt 改写、评分器调整和结论归纳，必须遵守完整原则。
