总判断

截至 2026-06，自动 prompt 优化（APO）领域已经从「找一个更聪明的改写算法」收敛到「建一套可审计的优化工程」：方法侧的真正分类轴是反馈信号的信息密度（标量 → 批评 → 完整轨迹），工程侧的入场券是五件套（prompt + dataset + metric + trace + constraints）。算法和基础设施缺一不可——论文给的是机制和数字，生产里跑得起来靠的是版本、trace、验证集隔离和回滚。

五个最重要的可执行判断

  1. 优化之前先体检。某些设置下 49% 的优化运行比 zero-shot 还差（Coin Flip，72 次运行，Haiku 4.5）；唯一让 6 种方法全部正收益的是结构化 JSON 输出任务。headroom 没超过噪声地板就别花钱。A
  2. 有 dev set 时，先优化示例，不是指令。MIPROv2 的分诊结论：多数任务优化 bootstrapped demonstration 比优化 instruction 更管用，联合优化在 7 任务中 5 个最优；instruction 优化只在「有隐藏条件规则」的任务上决定性。A
  3. 反馈信息密度决定样本效率。从标量（APE/OPRO）到失败批评（ProTeGi/TextGrad）到完整执行轨迹（GEPA），单位 rollout 榨取的信息持续上升——GEPA 用约 3,936 rollouts 超过用 24,000 rollouts 的 GRPO（RL）。选方法先看你的系统能产出什么信号。A
  4. 零成本结构变换是新的对照底线。把整个 prompt 原样重复一遍，在非推理模式下跨 7 个模型 47/70 显著胜、0 显著负——任何搜索式优化的增益都应先回答「打得过这个零智能变换吗」。Arecent
  5. 没有五件套就没有优化闭环。optimizer 的输入是 prompt + dataset + metric + trace + constraints，不是一段待润色的文本；缺 eval 时所谓优化只是风格化改写。这是四个工程渠道独立收敛的同一结论。B

最大风险

prompt 膨胀式过拟合（放任时 TextGrad 把 prompt 撑到 14.7 倍长、86% 在偷改任务定义）、reward hacking（optimizer 摸到 evaluator）、记忆污染（未过滤历史造成跨任务负迁移）、vendor 数字幻觉（任务/指标/展示偏向叙事）、以及跨模型脆弱性——同一套方法在 Haiku 上全胜的配置，换到 Nova Lite 上 24 个均值里 14 个低于 zero-shot。

选型导引：六个问题问完再选方法

  1. 有留出集和评分器吗？没有 → 先别选方法，去凑五件套（④ 工程落地）；缺 eval 时任何「优化」都只是风格化改写。
  2. headroom 体检过了吗？用 HM-01 跑体检（洞见 01）；headroom 没超过噪声地板 → 停。顺手先测零成本结构变换（洞见 13）当对照底线。
  3. 有 dev set？先跑示例优化基线（HM-03）：多数任务它比改 instruction 更有效（MIPROv2 / Teach Better）。
  4. 任务有「隐藏条件规则、少量样例表达不了」？是 → 才值得上 instruction 优化（MIPROv2 分诊结论）；否则示例 + 轻量指令微调即可。
  5. 系统能产出可解释 trace（agent / 工具 / 多模块）？能 → 反思式方法（GEPA 类）才有信息密度优势；只能给标量分 → 优势收窄，老老实实用 APE/OPRO 风格搜索。
  6. 复杂方法打得过下限基线吗？任何新方法的对照组至少含 APE（propose-then-select）与 OPRO（标量轨迹）；打不过就不值得引入。

动手之前：这个任务值不值得优化？

最容易被跳过、也最省钱的一步：在花钱跑优化器之前，先确认有没有提升空间、以及任务类型能不能让你看清效果。

洞见 01证据 A

先确认有没有提升空间，再花钱跑优化器

自动优化不是稳赚的——某些设置下，它有近一半概率让 prompt 比 zero-shot 还差。

一个例子（示意）

工单三分类，初版 prompt：
「把工单分到：投诉/咨询/退款」
30 条样本测：准确率 82%
优化器跑 6 轮 → 又长又"专业"的新 prompt
dev：87% ↑（看起来涨了）
100 条测试集：79% ↓ 比最初还低
那 5 分是小样本噪声里捡的

论文怎么说

49%Coin Flip：72 次优化里 49% 低于 zero-shot（Haiku 4.5）
~3 步ProTeGi：约 3 步达峰，之后多在训练集过拟合

来源：coin-flip-2026（recent-preprint）、protegi-2023；「在选择信号里加风险项也救不回来」的受控反例见 prose-2026

你该怎么做

  1. 跑 zero-shot + 手写 baseline，各记一分。
  2. 生成 10–20 个候选，用同一批 20 条留出样本打分。
  3. 看 headroom（最高候选 − zero-shot）和 noise floor（候选间抖动）。
  4. headroom 没超过 noise floor → 停，别优化。
跨渠道互证：web_search 的 WHM-01「Optimizer Intake Checklist」与 Ask HN「AI evals 半成品」痛点信号、Twitter tw-insight-01「先有测试集和评分器」、知乎 ZHI-01「从技巧转向 eval 驱动」均独立指向同一前置纪律。
边界：靠多轮搜索才能发现组合结构的任务会被这个体检低估；阈值要按样本量和评分器重校准。
洞见 02证据 A/B

第一批先选「能客观打分」的任务

任务类型决定你能不能看清优化有没有用。开放写作就算优化了，你也分不清是真变好还是评委口味变了。

只能靠另一个模型当评委打分
评委口味天天变 → 分数变化无法归因
任务乙：抽成 {intent, entities, urgency} JSON
漏抽 urgency → field F1 该列直接掉
失败能拆成 漏抽/标错/格式坏/过度推断

论文怎么说

+6.8Coin Flip：唯一让 6 种方法全超 zero-shot 的是结构化 JSON 任务 HelpSteer2

你该怎么做

  • 第一批验证选：信息抽取（JSON）、文本分类、工具调用。
  • 暂时别用开放生成当第一块试验田——因果最难看清。
跨渠道互证：对应统一目录 I-11；Twitter conclusion-01/02 把「有留出集、可分类失败、有可解释中间过程」列为方法成立前提；web_search WPI 也强调主观写作任务必须先定义 rubric，否则 metric 不稳。
边界：开放生成很重要，只是不适合作为「因果清晰」的首批验证。
③ 工作流洞见 Part B

怎么从失败里学到东西

分数告诉你「坏了」，文字诊断和根因假设才告诉你「该改哪里」。这是 prompt 改进真正的燃料。术语提醒：所谓「文字梯度」只是工程比喻，不是数学梯度——本报告统一按「反馈驱动改写」理解。

洞见 03证据 A证据升级

把失败样本变成「可编辑的证据」，不只是一个分数

accuracy 那个数字无从下手；一句具体的失败诊断才直接指向 prompt 该补什么。

一个例子（示意）

工单：「我要退货并且投诉客服态度」
模型抽出 intent: 投诉（漏了 退款）

只记分数 → 这条 = 错（一个红叉，没线索）
记成证据 → "当一条工单同时含『动作请求』和『情绪』时，
            模型只抓情绪、漏掉动作" ← 直接告诉你该补哪条规则

论文怎么说

+15.3% / 最高 +31%ProTeGi：用失败样本生成文字反馈再改写，比原始 prompt
0.760 vs 0.110SPEAR：给 optimizer 配 Python 工具做错误分析，难任务 kappa 大幅超 TextGrad

来源：protegi-2023、spear-2026（recent-preprint：错误分析可以用代码工具而不只靠长上下文）、textual-gradients-flawed-metaphor-2025（「梯度」是比喻，非真梯度）

你该怎么做

  • 每个失败 case 记：prediction、gold、error_type、critique、candidate_id。
  • 先把 20 条失败整理成「失败类型表」，再让优化器只基于这张表改。
  • 失败样本多时（>100 条），考虑让分析跑在代码里（pandas/groupby 聚合错误类型）而不是全塞进上下文——SPEAR 显示长上下文聚合不可靠。
跨渠道互证：GitHub 的 compare-first rewrite（linshenkx/prompt-optimizer 源码）保留失败证据与 diff；Twitter tw-insight-02「让优化器读失败过程，而不只读最终分数」；web_search WHM-03 与知乎 HM-ZH-02 都要求按失败 owner 分层诊断。
边界：critique 会带评委偏见，必须绑定样本和指标，不能当成真实因果。Flawed-Metaphor 的警告：文字反馈的提升常来自格式与 meta-instruction，甚至 prevalence hacking——把 critique 当线索，不当真理。
洞见 04证据 Arecent-preprint

先列「失败根因假设」，再改 prompt

反思式优化器最常见的翻车，是一开始就猜错方向，然后在错误方向上越改越自信。

一个例子（示意）

urgency 判错 30%。不要直接"让模型改好"，先列假设：
 A 缺判定标准   → 候选A：补 urgency 判定规则
 B 缺边界示例   → 候选B：加 2 个"看似急其实不急"的例子
 C 中性词误判   → 候选C：声明"尽快/马上"不单独构成紧急
三个候选都在 minibatch 验证，谁真把 30% 拉回来用谁

论文怎么说

23.81% → 13.50% → 87.57%VISTA：GEPA 在缺陷种子上反而掉到 13.50%；解耦"假设生成/改写"后恢复 87.57%

来源：vista-reflection-dark-2026、llm-prompt-optimizers-2024

你该怎么做

  • 对同一批失败，强制生成 2–3 个互斥根因假设（缺规则/缺示例/格式冲突）。
  • 每个假设生成一个候选，分别评分；运行记录存 failure_hypothesis 而不只 critique。
跨渠道互证：本条目前主要由 arXiv（VISTA）单层支撑，工程渠道尚无独立证据——属于「机制清晰、待工程与本项目验证」类，已列入 P1。
边界：开放写作任务根因模糊，多假设验证可能成本大于收益。
③ 工作流洞见 Part C

别让模型骗你：候选选择与防膨胀

LLM 候选方差极大。怎么"选"和怎么防"越改越长"，是把伪提升挡在门外的两道闸。

洞见 05证据 A

不要采用第一版，多生几个再用验证集选

「怎么选候选」和「怎么生成候选」一样重要。把选择权交给验证集，别让模型自评"我这版最好"。

一个例子（示意）：一张候选账

候选  dev分  格式错  长度   选不选
C1   0.86   0%    1.0x   ✔候选
C2   0.88   6%    1.2x   ✗弃（格式错升高）
C3   0.87   0%    2.4x   ✗弃（太长，疑似过拟合）
C5   0.86   0%    1.0x   ✔候选
不是只挑 dev 最高的 C2，而是综合三维选；best-seen 永不被直接覆盖

论文怎么说

0.85 / 0.67 / 0.88ProTeGi：beam 选择在 Jailbreak/Liar/Sarcasm 明显优于贪心；bandit > 均匀选
+12.44 vs +6.05GEPA 消融：Pareto 候选池本身是主要贡献，明显优于贪心选择

来源：protegi-2023、gepa-2026、promptbreeder-2023

你该怎么做

  • 每轮存 seed/candidates/scores/selection_policy/rejected_reason/best_seen/rollback_point。
  • 生成 5 个候选，按 dev 分 + 格式错误 + 长度三维选 Pareto 候选。
跨渠道互证：GitHub insight cards 把 candidate ledger 列为可复用结构（→ HM-04）；Twitter method-01「metric+trace 约束下生成并选择候选」对比 scalar-only 与 trace-aware 反馈，是同一原则的业界版。
边界：搜索预算越大越容易 dev 过拟合，必须有 validation/test 分层兜底。
洞见 06证据 A证据升级

prompt 卫生是主动优化指标：变长、变复杂往往是过拟合不是进步

优化后更长、规则更多、看起来更专业，很可能是在给训练样本打补丁。卫生（hygiene）不该只是事后检查，要进选择函数。

一个例子（示意）：优化器加的"伪规律"

- 若提到「发票」且含数字，intent 一定是退款   ← 某条样本的补丁
- 若出现「经理/投诉」，urgency 一律 high      ← 局部特例当全局规则
- 感叹号超过两个 → 倾向 high                 ← 在拟合标注者习惯
dev +5，压力测试集 −8

论文怎么说

14.7x / 86%PrefPO：放任时 TextGrad 撑到 14.7 倍长、86% 在偷改任务本身（PrefPO 仅 37%）
+10.0 / +9.9TextReg：加长度+范围正则化管住膨胀，反而比 TextGrad 更高

来源：prefpo-2026、textreg-2026、causal-edit-level-2026（均 recent-preprint；后者显示 meta-instruction、复杂化对数学/多跳任务有负关联——不同任务对编辑方向的反应是异质的）

你该怎么做

  • 把 hygiene 写进候选选择：length_ratio、repetition、edit_family、dev_test_gap、OOD/stress_delta 与 dev 分一起进 Pareto。
  • 候选若变长，必须说明每条新规则绑定哪些失败样本并通过压力测试，否则拒绝。
  • 过拟合不只是 dev-test gap：规则变窄（scope 收缩）和表示膨胀同样是信号（TextReg 的形式化）。
跨渠道互证：GitHub 与 web_search 的反模式表都把「只看平均分上升」单列（可能掩盖 schema 漂移、少数类回退、安全退化、成本膨胀）；web_search 另把「采信 vendor 提升 X%」列为同源风险。综述视角下这条线就是「约束优化」（constrained optimization）的实证版。
边界：有些复杂任务确实需要更明确约束——标准是「必要、可追溯、不过拟合」，不是「短」。
③ 工作流洞见 Part D

到底该改什么（不只是那段 prompt）

真实系统里失败可能来自示例、prompting 模式、工具描述——先标清你改的是哪个部件，否则无法归因。

洞见 07证据 A证据升级

有开发集时，选「给模型看的例子」常比改 instruction 更有效

大家本能去改 instruction 措辞，却忽略了常更有效的变量——给模型看哪几个示例。这条已从单篇支撑升级为多篇独立支撑。

一个例子（示意）

instruction 不动，只换 3 个精选示例（覆盖易错边界）：
 例1 动作+情绪并存：要退货还要投诉 → intent:[退款,投诉]
 例2 看似急其实不急：方便时帮我看下 → urgency: low
 例3 多实体：订单 123 和 456 都改地址 → entities:[123,456]
往往比绞尽脑汁重写 instruction 收益更大

论文怎么说

+12.63Teach Better：不改 instruction、只优化示例就达 72.92（PaLM 2 / BBH）
5/7MIPROv2：联合优化在 7 任务中 5 个最优；分诊结论——多数任务 demo 优化 > instruction 优化

来源：teach-better-show-smarter-2024、miprov2-2024、erm-memory-2024（exemplar factory）；instruction 与 demo 互补的证据见 textgrad-2024

你该怎么做

  • 至少比较 no-example / random-example / optimized-example / instruction+optimized。
  • 固定 instruction，从候选池搜 3 个示例看留出表现。
  • 分诊规则：任务含「隐藏条件规则、少量样例表达不了」时，instruction 优化才是决定性的（MIPROv2 HotPotQA Conditional 例证）。
跨渠道互证：对应统一目录 I-05 / HM-03，由 arXiv 多篇（Teach Better、MIPROv2、ERM、DistillPrompt）支撑；列入 P0 首批验证，正是为了防止把示例收益误记成指令收益。
边界：上下文紧张、示例分布偏、示例可能泄漏答案时，会伤害泛化。

先标清楚你改的是哪个「部件」

失败可能来自示例、prompting 模式、工具 schema、agent 角色、记忆或评委，不只是 system prompt。

洞见 09证据 Arecent-preprint

工具调错时，改的常是「工具说明」，不是 agent 指令

失败常来自工具描述和参数语义本身；改 agent 的 system prompt 再多遍也没用。

一个例子（示意）

"请正确填写金额"
→ 模型仍不知道单位，传 amount: 99.5

③ 工作流洞见 Part E

记忆与多 agent：更多 ≠ 更好

加记忆、加 agent 听起来总是好的，但未经筛选的记忆会污染优化，整体重写会让你无法归因。

洞见 10证据 A/B

只有「过滤后的记忆」才有用

未经筛选地堆历史，会造成跨任务负迁移，还更贵。

一个例子（示意）

无界记忆：把所有历史对话塞进上下文
→ 把 A 产品线的退款规则错套到 B 产品线工单
过滤记忆：每条带元数据
 content / source_task / applicability
 quality_score / retrieval_reason / forget_after
只取 3–5 条已验证、且适用条件匹配的

论文怎么说

70.7% vs 63.6%｜$0.31 vs $0.70MemAPO：双记忆（成功模板+错误模式）比 TextGrad 更好且更便宜
第7步 68.6 vs 第13步 58.5ERM：过滤记忆收敛近两倍速；原始记忆无收益

  • 记忆 schema 含 success_template/error_pattern、source_task、applicability、quality、forget_reason。

边界：非平稳/安全敏感任务里，遗忘策略和 opt-out 比「记得多」更重要。
洞见 11证据 Arecent-preprint证据升级

多 agent 出错，先找「是谁、在哪一轮」的责任，别整个重写

核心难题不是加更多 agent，而是 credit assignment——而且它有两个维度：结构（哪个角色）和时间（哪一轮）。

一个例子（示意）

抽取 → 补全 → 回复，最终回复错了。看 trace：
 agent_1 抽取  local_valid ✗  ← urgency 在这里就丢了
 agent_2 补全  local_valid ✓  （基于错误输入 = local_pass_global_fail）
 agent_3 回复  local_valid ✓
根因在 agent_1，只改 agent_1，别动另两个

论文怎么说

55.13 → 64.63Temporal/Structural Credit：credit-guided 在 MedMCQA（+9.50）；无差别更新所有轮反而掉分
70.39 vs 65.31MASPO：联合优化平均优于顺序 baseline；专门收集「局部成功、全局失败」的错配样本

来源：temporal-structural-credit-mas-2026（时间轮次 × 结构角色双轴分解 + 交替更新）、maspo-2026、mapro-2025（沿拓扑传播 credit）

你该怎么做

  • trace 存 role_id/round_id/local_validity/successor_utility/global_outcome——时间和结构两个维度都要。
  • 标出首次引入错误的 role/round，只改那一块；耦合弱时先做 coupling test。
跨渠道互证：本条主要由 arXiv（MASPO / MAPRO / Temporal-Structural Credit）单层支撑，三篇互补不矛盾（拓扑 / 错配检测 / 时间-结构分解），工程渠道尚无独立证据；与洞见 08 的「先标责任部件、再局部改」属同一治理思路。
边界：agent 交互很弱时，联合优化成本可能不划算——先测耦合再决定。
③ 工作流洞见 Part F

怎么对待「听来的」方法

热度只说明传播，不说明有效。二手传播会放大结论、丢掉边界和版本。这一条把上面几层的渠道纪律收口；完整的误区清单见 ⑥ 误区澄清。

洞见 12证据 B

社媒和二手文章是「线索」，不是「证据」

一个方法被反复转发，只能说明它热，不能说明它有效。

一个例子（示意）

爆款：「GEPA 用少 35 倍 rollout 吊打 RL！」
当证据 → 直接信「GEPA 能替代 RL」
当线索 → 翻原论文发现：
GEPA 是反思式 prompt 进化（执行轨迹 + Pareto 选择），
根本不是 RL 系统；AIME-2025 上仍低于 GRPO；
"替代 RL"是传播简化出的口号

你该怎么做

  • 读到技巧先抽六字段：dataset / metric / baseline / model / cost / failure_cases。
  • 缺任一字段就降级为「线索」，进结论前必须追溯论文/官方文档/代码/本项目实验。

来源：twitter_web_analysis_20260608、web_search_platform_analysis_20260608

跨渠道互证（本条由社媒/二手渠道共同提出）：Twitter 把它做成 method-05 social_signal_triage（六类标签）与 method-06 vendor_optimizer_evidence_filter；web_search 做成 WHM-04 source triage；知乎做成净新增 HM-ZH-03「一手来源升格门」。三个渠道独立收敛出同一道闸。
边界：自带代码、数据、可复现实验的高质量工程博客可升级为更强证据——看细节，不看平台。
③ 工作流洞见 Part G · 本版新增

搜索之外：零成本变换与「优化器自身」

v3 的 12 条洞见全部围绕「搜索/优化循环」构建。各渠道这一轮深挖暴露了两个循环之外的盲区：一个是不用搜索的确定性结构变换（比所有搜索方法都便宜、却常被跳过的对照），一个是循环里被当成空气的 optimizer 与 judge 本身。

洞见 13证据 Arecent-preprint本版新增

花钱搜索之前，先试零成本的确定性结构变换

一个不改任何语义的固定变换——把整个 prompt 原样重复一遍——在非推理模式下跨 7 个模型 47 胜 0 负；而搜索式优化有近一半概率比 zero-shot 还差。

一个例子（真实任务，论文数字）

变换：<QUERY>  →  <QUERY><QUERY>（原样重复，无任何措辞修改）

NameIndex（50 个名字取第 25 个）：
Gemini 2.0 Flash-Lite  21.33% → 97.33%
机制：causal attention 下 prompt 前段 token 看不到后段；
重复后第二遍的每个 token 都能注意到完整第一遍

论文怎么说

47/70 胜 0 负非推理模式，7 模型 × 10 配置，McNemar p<0.1；Padding 对照无效→收益来自重复本身
5/28开启 step-by-step 推理后基本无效（推理模型本就常自发复述题目）

来源：prompt-repetition-2025（Google Research；不增加输出长度与时延，重复发生在可并行的 prefill 阶段）

⑥ 误区澄清：12 个最常见的误读

知乎与 Twitter 渠道的独特价值不是证明什么，而是暴露「大家正在怎么误解」。这里把两渠道收敛的误区合并去重成 12 条——每一条都在真实传播中反复出现。

#误读实际情况出处
1「GEPA 全面超过 / 替代 RL」GEPA 是反思式 prompt 进化（trace + Pareto），不是 RL 系统；AIME-2025 上仍低于 GRPO；优势依赖 feedback 质量，评价只能给标量时优势收窄。知乎 ZHI-02 / Twitter conclusion-02 / arXiv A 级
2「DSPy 是自动改提示词工具」DSPy 是编程框架（signature/module/metric 都要手工定义）；2023 版优化的是 demonstration，instruction 自动优化要到 MIPROv2。知乎 ZHI-03 / arXiv dspy/miprov2
3「文字梯度 = 梯度下降」是工程比喻：不可加、不可组合、错误标签常不伤害结果；提升常来自格式与 meta-instruction，甚至 prevalence hacking。arXiv Flawed-Metaphor（2025-12）
4「让模型优化一下 prompt 就行」没有留出集和评分器时，只能得到「更像 prompt 的文本」，无法判断变好变坏。Twitter conclusion-01 / 全渠道
5「优化后更长更专业 = 更好」多为训练样本补丁；放任时 TextGrad prompt 长 14.7 倍、86% 在偷改任务定义。arXiv PrefPO/TextReg（洞见 06）
6「context engineering = 写更长 prompt」是检索/记忆/工具输出/历史压缩的分变量工程；prompt 优化只是其严格子集。知乎 ZHI-04 / Context Engineering 综述
7「自进化 = 什么都记下来」未过滤记忆造成跨任务负迁移还更贵；每条记忆要有来源、适用范围、验证与遗忘策略。arXiv ERM/MemAPO / 知乎 / GitHub ECC
8「转发多 / star 多 = 方法有效」热度只说明传播；多个二手来源指向同一论文时，证据强度由原论文决定。全渠道（洞见 12）
9「厂商提升百分比 / demo 截图 = 证据」任务、指标、展示都可能偏向产品叙事；只采流程字段，数字默认降级。web WPI-07 / Twitter conclusion-05
10「仓库有测试 = 方法有效」tests 可能只覆盖安装/格式/配置；「可审计」与「有效」是两个主张，分开记。GitHub GHI-11
11「prompt 技巧跨模型通用」OPRO 搜出的钥匙短语高度 model-specific；Haiku 上 6/6 全胜的配置在 Nova Lite 崩到 1/6。arXiv OPRO / coin-flip（① 方法地图）
12「一次多改几个变量更快」prompt/context/model/evaluator 同时改 → 指标变化无法归因，形成伪结论；一次只改一层。全渠道 / CLAUDE.md 同款纪律

刻意边界：我们选择不做什么

  • soft / continuous prompt tuning（prefix/embedding 级)——聚焦离散自然语言 prompt，评分规则显式排除。
  • multimodal（text-to-image/video、VLM prompt 优化）——两份综述都列为方向，本项目明确不做。
  • 完整 context-engineering 系统（RAG 架构、检索器优化、多 agent 通信协议）——当边界材料；但「上下文压缩」与「智能上下文组装」两条与 prompt 优化同构，已登记 backlog。
  • 这些是范围选择，不是疏漏——依据 taxonomy 完整性校验（对照 3 篇综述）。

frontier 缺口：领域有、我们还没有格子的方向

  • bi-level / thought-driven prompt 优化（推理模型 o1/R1 时代）——唯一「整块相邻且完全缺席」的方向，下一次定向 arXiv 搜索的最高优先级。
  • task-agnostic / online 优化（无 Dval、意图漂移、非平稳）——与自进化主线相关，值得一次定向搜索。
  • constrained optimization——本报告的 hygiene 主题（洞见 06）就是它的实证版，理论外壳待补。
  • multi-task / negative transfer——与记忆过滤（洞见 10）合并讨论。
  • 簇内偏轻子机制（学习型评分、token 级编辑、bandit 筛选、MoE 路由）——已知存在、刻意不深入，仅留指针。

⑩ 把这些用起来：首批最小验证

最值得本项目先动手验证的几条——成本低、错误可解释、跨渠道支撑强。详细设计见 实验计划，与统一目录的 P0–P2 同口径；本版新增「零成本变换三臂 A/B」为最便宜的 P0。

给读者的四条总结：① 优化前先问「值不值得」，headroom 没超过 noise floor 就别花钱，顺手先测零成本的结构变换当底线；② 选方法看你的系统能产出什么反馈信号（标量 / 批评 / 轨迹），有 dev set 先优化示例而不是指令；③ 改 prompt 的本质是「证据 → 假设 → 多候选 → 验证集筛选 → 可回滚」，不是让模型润色；④ 更长、更多、更复杂都不是默认的好——先证明它没带来过拟合、污染或归因困难。
