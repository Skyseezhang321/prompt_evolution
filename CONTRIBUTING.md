# 参与贡献指南

欢迎参与 `prompt_evolution`。这个项目研究 prompt 优化与 prompt 自进化，公共共创的目标不是把所有观点都写进结论，而是把有价值的阅读发现转成可追踪、可验证、可回滚的研究线索。

核心规则：

> 任何人都可以低成本提交研究线索；任何项目结论都必须经过新颖性判断、证据整理和可追踪验证。

## 你可以贡献什么

| 贡献类型 | 适合场景 | 推荐入口 | 最低要求 |
| --- | --- | --- | --- |
| 研究线索 | 读到一篇论文、文档、thread、issue 或事故复盘，觉得某个点有帮助 | `Research Signal` issue | 来源链接、你觉得有价值的点、可能关联的项目问题 |
| 深读笔记 | 愿意系统总结一篇论文或行业来源 | PR + 结构化笔记 | 使用 `docs/paper_notes/template.md` 或 `docs/industry_notes/template.md` |
| 实验提案 | 某个观点可以转成最小 eval 或 prompt 实验 | `Experiment Proposal` issue | 目标、假设、输入样本、模型、评估方式、成功标准 |
| 实验结果 | 已经跑出可复查的指标、样例或失败案例 | PR | 运行记录、prompt diff、模型参数、数据集、评分器、成本、失败案例 |
| 原则沉淀 | 多个来源或实验支持同一条稳定经验 | PR | 明确证据来源，区分观察、推测和结论 |

## 最快参与方式

1. 打开一个 `Research Signal` issue。
2. 填写来源、关键观点、为什么你认为它有帮助。
3. 如果你知道它和项目已有内容的关系，可以写上“可能是重复、补充、冲突或新假设”。
4. 维护者或 reviewer 会做项目内新颖性判断，并决定进入深读、实验候选、补充引用或关闭。

不需要一开始就写完整论文笔记。低门槛线索是有效贡献，但线索本身不会直接变成项目结论。

## 分支与合入要求

本轮流程规则变更完成提交并推送后，后续所有代码、文档、实验和配置改动都必须先从最新 `main` 创建个人或任务分支，在自己的分支内修改、验证并提交；不得直接向 `main` 提交。

这条规则的目的，是把 `main` 保持为可复现、可审查、可回滚的稳定基线，同时把不同贡献者、不同假设和不同变量的改动隔离开。这样 reviewer 可以按 PR 判断影响范围，必要时也能按分支或提交快速回滚。

提交合入时请通过 PR/合入请求申请进入 `main`，并说明分支来源、改动范围、验证结果、是否影响 prompt/数据/模型参数/评分器，以及回滚点。工作区里如果有无关改动，不要混入本次 PR。

## 新颖性如何判断

这里的新颖性是“项目内新颖”，不是判断世界范围内是否首创。

| 标记 | 含义 | 后续处理 |
| --- | --- | --- |
| `duplicate` | 项目已有类似观点或来源 | 合并引用，关闭或链接到已有 issue |
| `extension` | 对已有观点提供边界、案例、指标或反例 | 补充到相关笔记或综述 |
| `contradiction` | 与已有判断冲突 | 标记为需要证据，优先进入讨论或实验候选 |
| `new-hypothesis` | 形成新的可检验假设 | 进入深读或实验提案 |
| `actionable-experiment` | 可以直接转成最小实验 | 先更新 `docs/experiment_plan.md`，再实现 |

## 写笔记时的要求

- 新论文笔记放在 `docs/paper_notes/`，使用 `docs/paper_notes/template.md`。
- 新行业经验放在 `docs/industry_notes/`，使用 `docs/industry_notes/template.md`。
- 所有来源先登记到 `docs/source_inventory.md`，再决定是否进入深读。
- 明确区分作者主张、可观察证据、本项目推测和已验证结论。
- 不要把单次样例表现写成稳定规律。
- 对论文、文档、博客、thread 和 issue 保留原始链接。

## 写实验时的要求

新实验必须先更新 `docs/experiment_plan.md`，再实现脚本或运行实验。实验记录至少包含：

- 目标和假设。
- baseline prompt 和候选 prompt。
- 模型、参数、数据集、评分器。
- prompt diff、成本、失败案例和回滚点。
- 结论限制：哪些只是观察，哪些仍待验证。

## PR 要求

提交 PR 时请说明：

- 对应的 issue、来源或实验假设。
- 改了哪些文件，以及为什么这些文件需要改。
- 是否改变了 prompt、数据、模型参数、评分器或分析口径。
- 是否更新了 `CHANGELOG.md` 的 `Unreleased`。
- 证据来自哪里：来源笔记、运行记录、指标表、失败案例或反例。

## 安全、隐私和版权

- 不要提交 API key、webhook、账号 token 或私有数据。
- 真实 webhook 只能放在本地 `.env`，不得提交。
- 不要上传未经脱敏的用户对话、客户数据、内部日志或商业机密。
- 对知乎、Twitter/X、付费文章等来源，只保留必要短摘录、链接和摘要，不要整篇转载。

完整共创流程见 [共创工作流](docs/contribution_workflow.md)。
